1 · Fundamentals & IS Approaches

Lectures 1–2 · Exam Q1

1.1 Key Definitions 🔴

Information Technology (IT): Use of hardware, software, services, and supporting infrastructure to manage and deliver information (voice, data, video).
System: A set of interrelated components that work together to accomplish a common goal.
Information System (IS): An arrangement of people, data, processes, and technology that collects, processes, stores, and outputs the information an organization needs.

IS has 4 components:

  • People — users, IT staff, managers
  • Data — raw facts collected and processed
  • Processes — activities that transform data into information
  • Technology — hardware, software, networks
Q: What is the PRIMARY objective of SA&D?
A: NOT to create a wonderful system — it is to create VALUE for the organization (increase profits, improve processes, serve users better, integrate with other IS).
⚠️ Common Mistake: Students say "the goal is to build a great system." Wrong. The goal is to create business value. Many projects fail because they focus on the system, not the business.

1.2 IS Types 🟡

Abbreviation Full Name What it does Level
TPS Transaction Processing System Handles day-to-day repetitive transactions (sales, payroll) Operational
MIS Management Information System Generates summary reports from TPS data Middle Management
DSS Decision Support System Supports semi-structured decisions using models Senior Management
ES Expert System Emulates human expert knowledge; gives recommendations Specialist

1.3 Roles of the System Analyst 🔴

Exam Q1-B asks for 4 of the 6 roles. Know all 6.

  1. System Analyst — Studies org. problems and designs IS solutions
  2. Business Analyst — Focuses on business problems, independent of technology
  3. Systems Architect — Designs the overall technical architecture
  4. Infrastructure Analyst — Focuses on networks, hardware, OS
  5. Change Management Analyst — Helps people adapt to the new system
  6. Project Manager — Plans, executes, and controls the development project
📌 Exam Focus: Know all 6 roles by name and be able to describe any one in 2–3 lines.

1.4 Project Sponsor 🔴

Project Sponsor: The person who initiates the project. They identify the business need, define requirements, and prepare the System Request.

System Request has 5 elements:

  1. Project Sponsor
  2. Business Need
  3. Business Requirements
  4. Business Value
  5. Special Issues and Constraints
Q: Who is the Project Sponsor and what do they do?
A: The person who starts the project. They identify business needs and expected value, then formally prepare the System Request.

2 · SDLC & Methodologies

Lectures 3–4 · Exam Q2

2.1 SDLC Definition 🔴

SDLC (System Development Lifecycle): A logical process by which analysts, engineers, programmers, and users build IS to solve business problems. It is a project management tool used to plan, execute, and control development.

SDLC provides:

  • High-quality, low-cost software
  • Shortest possible production time
  • Minimizes project risks
  • Complete and consistent documentation
⚠️ Common Mistake: "SDLC is a methodology." Wrong. SDLC describes the phases. A methodology is how you execute those phases. (Fact: 70% of IT projects fail due to lack of SDLC implementation.)
Q: Is SDLC a methodology? Explain.
A: No. SDLC describes what phases to go through. A Methodology describes how to execute them.

2.2 The 7 SDLC Phases 🔴

  1. Planning — Identify project, feasibility analysis, steering committee approval
  2. Analysis — Three sub-phases: Survey → Detailed Study → Definition
  3. Design & Prototyping — Design architecture, inputs, outputs, interfaces, data storage, programs
  4. Implementation — Code, test, and integrate the software
  5. Configuration & Procurement — Define and acquire hardware/software/networks
  6. Deployment & Operations — Go-live, user training, acceptance testing
  7. Maintenance — Ongoing repair, updates, improvements
📌 Exam Focus: Memorize all 7 phases in order. Know that the Analysis phase has 3 sub-phases.

2.3 Project Payments Milestones 🟡

Project Payments Milestones: Checkpoints in a project timeline, each linked to a deliverable. Payment is released only after the deliverable is accepted. Ensures the client pays based on actual progress, not just time.

2.4 Methodologies Compared 🟡

Methodology Key Idea Best For
Waterfall Sequential, phase-by-phase, no going back Stable, well-defined requirements
Prototyping Build a working model to verify requirements Unclear requirements
Iterative / Incremental Deliver in pieces, each adds functionality Large systems with phased delivery
Spiral Iterative + risk analysis at each loop High-risk, complex projects
Agile Flexible, short sprints (≈2 weeks), collaborative Rapidly changing requirements
OOAD Object-oriented analysis and design OO-based systems

2.5 Feasibility Analysis (3 Types) 🔴

  • Technical feasibility — Can we build it with available technology?
  • Economic feasibility — Will it provide enough business value to justify cost?
  • Organizational feasibility — If we build it, will people actually use it?

2.6 Eight Underlying Principles of System Development 🟢

  1. Get owners and users involved
  2. Use a problem-solving approach
  3. Establish phases and activities
  4. Establish standards
  5. Justify system as a capital investment (ROI)
  6. Do not be afraid to cancel or revise scope
  7. Divide and conquer
  8. Design for growth and change

3 · Analysis Phases

Exam Q5 — 30 marks (highest weight question)

The Analysis phase has 3 sub-phases:
1. Survey Phase → 2. Detailed Study Phase → 3. Definition Phase

3.1 Survey Phase 🔴

Survey Phase: The first sub-phase. Its goal is to quickly understand the current system's problems, opportunities, and constraints — and to plan the project.

4 Activities in Survey Phase:

  1. Survey Problems and Opportunities
  2. Negotiate Project Scope
  3. Plan the Project
  4. Present Project (to management for approval)
Q: What are the objectives of "Plan the Project" in Survey Phase? (Exam Q5-A)
A: (1) Select a project methodology, (2) Create a detailed work plan, (3) Identify staffing requirements, (4) Prepare to manage and control the project.
Deliverables: work plan, staffing plan, standards list, project charter, risk assessment plan.
Q: Why assign a Planner in the Survey Phase? (Exam Q5-B)
A: Because this phase requires creating a comprehensive project work plan, estimating costs/resources, scheduling activities, and identifying risks — specialist planning skills needed before committing full resources.

3.2 Detailed Study Phase 🔴

Detailed Study Phase: The second sub-phase. Analysts deeply investigate the existing system, gathering detailed information to understand all business processes, problems, and requirements.

6 Activities:

  1. Model the Current System (using DFDs)
  2. Analyze the Business Processes
  3. Analyze Problems and Opportunities
  4. Establish Systems Improvement Objectives and Constraints
  5. Modify Project Scope and Plan
  6. Present Findings and Recommendations
Q: What is Prototyping? Name 3 characteristics. (Exam Q5-D)
A: Building a working model of user requirements to discover and verify them before the full system is built.
Characteristics: (1) Iterative — build → evaluate → refine cycle, (2) User-centric — users actively provide feedback, (3) Preliminary — it is NOT the final system.
Q: Why should the PM prioritize business requirements? (Exam Q5-E)
A: Resources (time, budget, staff) are always limited. Prioritization ensures the most critical, high-value requirements are done first, allows incremental delivery, and reduces project risk.

3.3 Definition Phase 🔴

Definition Phase: The final sub-phase. The analyst defines requirements for the proposed new system and documents the system proposal.

4 Activities:

  1. Define proposed system requirements (functional and non-functional)
  2. Develop alternative solution approaches
  3. Evaluate and recommend the best alternative
  4. Transition to the Design Phase
Q: Why assign a Modeler in the Definition Phase? (Exam Q5-F)
A: This phase requires creating formal system models (DFDs, use cases, data models). Modeling specialists ensure models accurately communicate the proposed system to all stakeholders before design begins.

3.4 Types of Requirements 🟡

Type What it covers Example
Business Requirements Overall goals; contributions to org. success "Produce performance reports"
User Requirements Usability, UI expectations of end users Easy navigation
Functional Requirements What the system must do "User authentication", "Reporting"
Non-Functional Requirements Characteristics the system must have "Response time < 2 seconds"

4 · Fact-Finding Techniques (FFT)

Lecture 7 · Exam Q3

Before starting, prepare an Investigation Plan (includes Terms of Reference + timetable). Must be approved by top management.

4.1 The 5 FFTs 🔴

  1. Interviewing
  2. Questionnaires
  3. Observation
  4. Sampling & Record Inspection
  5. Joint Application Development (JAD)

4.2 Interviewing 🟡

Interviewing: The most productive FFT. A structured conversation between the analyst and staff to gather information about the current system.

3 Steps: Plan → Conduct → Terminate

  • Planning: Define objectives, prepare questions, identify interviewees
  • Conducting: Ask questions, distinguish facts from opinions, use simple language
  • Terminating: Summarize information, allow follow-up if needed

Rule: Always use two interviewers — one to ask, one to record.

Ask Top Management about… Ask Operational Staff about…
Policies, strategies, future direction, constraints (budget, manpower, time) Functions performed, forms used, communication lines, work volumes, improvement suggestions

4.3 Questionnaires 🔴

Questionnaire: A set of written questions distributed to many people simultaneously to gather information efficiently.

When to use questionnaires (5 conditions):

  1. Large number of people must be surveyed
  2. Respondents are geographically dispersed
  3. Quantitative data is needed (statistics, frequencies)
  4. Anonymity is required for honest answers
  5. Opinions need to be validated from a wide group
Advantages ✅ Disadvantages ❌
Reaches many people quickly · Low cost · Anonymity encourages honesty · Easy to tabulate Low response rate · No follow-up questions · May be misunderstood · Shallow depth

4.4 Observation 🟡

Observation: The analyst directly watches how work is performed to understand actual (vs. stated) procedures.
  • ✅ Reveals actual vs. documented procedures · Firsthand data · Good for timing/volumes
  • ❌ Hawthorne Effect — people change behavior when watched · Time-consuming
Q: What is the Hawthorne Effect?
A: People change their behavior when they know they are being observed, making observations less accurate.

4.5 Sampling & Record Inspection 🟢

Analyzing a representative subset of existing documents (forms, reports, procedure manuals, organization charts, transaction logs) to understand data volumes and current processes.

5 · System Modeling & DFDs

Lecture 9 · Exam Q4 — 20 marks

5.1 What is System Modeling? 🔴

System Modeling: Developing and presenting an abstract blueprint of a system using graphical notation. It emphasizes the flow of information between modules.
Q: What are the 2 purposes of modeling techniques? (Exam Q4-A)
A: (1) Understand the current system — identify strengths and weaknesses. (2) Communicate with stakeholders — provide a clear visual for non-technical people.

5.2 DFD Symbols 🔴

Symbol Shape Meaning Example
External Entity Rectangle / Square Source or destination outside the system Customer, HQ
Process Circle / Oval Transforms data Validate Order
Data Store Open-ended rectangle Data at rest (stored data) Archive, Customer DB
Data Flow Labeled Arrow → Movement of data between components Requested Order

5.3 DFD Levels 🔴

Context-Level DFD (Exam Q4-B)

The highest-level view. Shows the entire system as ONE single process surrounded by all external entities. Shows only what goes in and out. No data stores, no internal processes shown.

📌 Rules for Context-Level DFD: One process bubble · All external entities around it · All data flows in/out shown · No data stores.

Level-0 DFD (Exam Q4-C)

Expands the single process into the major sub-processes (e.g., departments). Shows main processes, data flows between them, major data stores, and external entities. Processes numbered 1.0, 2.0, 3.0…

Level-1 DFD (Exam Q4-D)

Explodes each Level-0 process into its sub-processes (child diagrams). Each department gets its own Level-1 diagram. Sub-processes numbered 1.1, 1.2, 1.3…

Level-2 DFD

Explodes each Level-1 process further. Only created when a Level-1 process is still too complex to implement directly.

Q: What is the DFD Balancing Rule?
A: Data flows going into and out of a parent process must exactly match the data flows in the child diagram of that process.

5.4 Common DFD Errors 🔴

  • Black Hole — Process has input but no output
  • Miracle — Process has output but no input
  • Store-to-Store flow — Direct flow between two data stores (must go through a process)
  • Entity-to-Entity flow — Direct flow between two external entities
  • Unbalanced DFD — Child flows don't match parent flows
📌 Exam Focus: Q4 = 20 marks. Practice drawing Context, Level-0, Level-1 DFDs for the RRG Restaurant example. Know the 5 DFD errors.

6 · Technical Architecture (STA)

Lecture 5 · Exam Q2

6.1 STA Definition 🔴

System Technical Architecture (STA): The fundamental functional organization of a system — its components, their relationships, and the principles guiding design. Defines technologies used for data, processes, interfaces, and networks.

4 Software Functions (know these):

  1. Presentation Logic — User interface display and input
  2. Application Logic — Business rules, processes
  3. Data Access Logic — How data is accessed
  4. Data Storage — Where data is stored

6.2 Architecture Types 🔴

Client-Server Architecture

Processing is split between client and server. Client handles presentation; server handles data. Three tier types:

  • 2-Tier: Client (presentation + business logic) ↔ Server (data storage + DB logic)
  • 3-Tier: Client (presentation) ↔ App Server (business logic) ↔ DB Server (data)
  • n-Tier: Client ↔ Web Server ↔ App Server ↔ DB Server

Thick/Fat Client: Client has most of the application logic (less server-dependent).
Thin Client: Client has minimal logic (e.g., browser); server does most work.

Server-Based Architecture

The server performs ALL FOUR functions. Clients are "dumb terminals" that only display output. Simple but not scalable.

Q: What is the key difference between Client-Server and Server-Based?
A: In Client-Server, processing is split. In Server-Based, the server does everything and clients are just display screens.

6.3 System Acquisition Strategies (3) 🔴

  1. Custom Development (Build) — Build from scratch. Full control. Most expensive and time-consuming.
  2. Commercial Off-the-Shelf (Buy / COTS) — Purchase existing software. Faster and cheaper. Less customizable.
  3. Outsourcing — Hire an external company. Reduces internal workload. Organization loses some control.

7 · Input / Output / Interface Design

Exam Q3, Q4, Q5

7.1 Input Design Objectives (2 Objectives) 🔴

  1. Ease of Use — Make input completion easy and efficient
  2. Accuracy — Assure accurate completion through controls and validation

How to achieve Ease of Use:

  • Capture only variable data (data that changes each transaction)
  • Capture only base data (not data that can be calculated/derived)
  • Use checkboxes, dropdowns — minimize handwriting
  • Include instructions on forms
  • Sequence data top-to-bottom, left-to-right

Input Controls:

External Controls (UI controls) Internal Controls (Validation)
Text boxes · Radio buttons · Checkboxes · List boxes · Combo boxes · Spin boxes Completeness check · Range/Limit check · Combination checks · Self-checking digits (checksum, NID) · Picture checks (format)

7.2 Output Design Objectives (4 Objectives) 🔴

  1. Deliver appropriate quantity — Give each user exactly what they need
  2. Output reaches the right person — Design a distribution plan
  3. Output provided on time — A delayed decision support report is useless
  4. Choose the right output method — Print, screen, web, audio, etc.

7.3 Interface Design — Standard Options & Consistency 🔴

Q: What does "Standard Options & Consistency" mean and why is it required? (Exam Q5-A)
A: All screens in the system should look and behave the same way. Required because: (1) Users learn faster, (2) Reduces errors, (3) Reduces training time and costs, (4) Makes the system feel professional.

7.4 Interface Types (7) 🟡

Type Description Example
Natural Language User types/speaks naturally AI chatbots
Question-and-Answer System asks, user answers Wizard setup
Menu Interface User selects from options Main menu, dropdowns
Form-fill Interface User fills in fields Registration forms
Command-Language User types commands Command prompt, SQL
GUI Icons, windows, mouse-driven Windows, macOS apps
Other (Touch, Voice) Alternative input methods Touchscreens

7.5 Caption Types 🟢

A caption is a label for a field in a form. Types: Line · Above-line · Below-line · Boxed · Vertical check-off · Horizontal check-off · Table caption.

⚠️ Consistency Rule: Never mix caption types within the same form.

8 · Database Design

Lecture 9 (DB) · Exam Q6

8.1 Data Storage Objectives (3) 🔴

  1. Making data available — Data must be accessible when the user needs it
  2. Assuring data integrity — Data must be accurate and consistent
  3. Supporting efficient storage, retrieval, and update — Data must be quickly and easily accessible for decisions

8.2 Flat File vs. Database Approach 🔴

Feature Flat File Database Approach
Storage Simple 2D tables, each app has its own files Centralized, shared by many applications
Managed by Operating system DBMS (managed by DBA)
Data independence No — changing file structure breaks programs Yes — programs independent of structure

Flat File Disadvantages (know all 5):

  1. Data Separation and Isolation — Difficult to merge data across files
  2. Data Duplication — Same data stored multiple times → inconsistency
  3. Data Dependence — Programs break if file structure changes
  4. Poor Data Security — Hard to control access at field level
  5. Concurrency Problems — Multiple users updating same file causes conflicts

8.3 Database Approach Advantages (5) 🔴

  1. Eliminates Data Redundancy — Each fact stored once; all apps share it
  2. Data Independence — Changing the database structure doesn't break programs
  3. Improved Data Integrity — Rules enforced uniformly across all applications
  4. Better Data Security — Fine-grained access control by DBA
  5. Data Sharing — Multiple users/apps access same data simultaneously

8.4 DBA — Roles & Risks 🔴

DBA (Database Administrator): Specialist responsible for designing, implementing, maintaining, and securing the database.

2 Key Roles of the DBA:

  1. Database Design & Implementation — Creates tables, indexes, relationships
  2. Security Administration — Controls who can access what data
⚠️ Potential Harm a DBA Can Cause: (1) Delete/corrupt critical data, (2) Leak confidential information, (3) Create unauthorized backdoor accounts, (4) Deny access to legitimate users.

8.5 File Organization Methods 🟡

Organization Best When…
Sequential Hit rate is very high (process most records each time)
Indexed Hit rate is not high; need to find specific records
Indexed Sequential (ISAM) Hit rate is medium
Direct / Hashed Hit rate medium or low; need fast random access
Q: What file organization is best when hit rate is very high?
A: Sequential — when you process most records each run, sequential is most efficient.

9 · Program Design

Lecture 10 · Exam Q7

9.1 Modular Programming 🔴

Modular Programming: A design technique that separates program functionality into independent, interchangeable modules. Each module executes only one aspect of the functionality.

Advantages (know at least 3):

  1. Critical interfaces are tested first
  2. Multiple programmers can work simultaneously
  3. Code reuse — a module written once can be used many times
  4. More control over development — problems isolated to specific modules
  5. Makes identifying structure easier (through the structure chart)

9.2 Top-Down vs. Bottom-Up Strategy 🟡

Top-Down Bottom-Up
Starts from Highest-level (main) module Lowest-level modules
Direction Decomposes downward into sub-modules Groups basic modules upward
Best for New system design Reusing existing components
Structure Chart: A hierarchical diagram showing all program modules, their parent-child relationships, and the data that passes between them (data-in, data-out). Derived from structured design.

9.3 Cohesion & Coupling 🔴

Q: Define Cohesion and state the goal.
A: Cohesion = how focused a module is on a single task. Goal: MAXIMIZE cohesion (HIGH cohesion = module does one thing well = better quality).
Q: Define Coupling and state the goal.
A: Coupling = how much modules depend on each other. Goal: MINIMIZE coupling (LOOSE coupling = modules nearly independent = better quality).
Definition Goal Good = ?
Cohesion How focused a module is on one task MAXIMIZE HIGH Cohesion
Coupling How much modules depend on each other MINIMIZE LOOSE Coupling
📌 Memory trick: HIGH Cohesion + LOOSE Coupling = Good Quality Program Design.
