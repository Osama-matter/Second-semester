SDLC, Analysis Approaches & Business Requirements

Exam Q2 · 12 Marks · SDLC · phases · business requirements · analysis strategies

SDLC Definition Must Memorize

System Development Life Cycle (SDLC): A logical process by which analysts, engineers, programmers, and users build information systems to solve business problems. It is a project management tool to plan, execute, and control development.
SDLC is NOT a methodology. SDLC = what phases. Methodology = how to execute them.

3 Key SDLC Phases Must Memorize

  1. Planning — Identify project; feasibility analysis; steering committee approval
  2. Analysis — Understand current system; gather requirements (sub-phases: Survey → Detailed Study → Definition)
  3. Design & Prototyping — Design architecture, inputs, outputs, interfaces, data storage, programs

Business Requirements Must Memorize

Business Requirements: Overall goals and capabilities an IS must have to contribute to organizational success. Describes what the business needs, not how it is implemented.
Two types

Fact-Finding Techniques & System Modeling (DFDs)

Exam Q4 · 15 Marks · Highest weight — master DFD drawing

Observation — When to Apply Must Memorize

Observation: The analyst directly watches how work is actually performed (vs. documented) to collect firsthand data on processes, volumes, and timing.
  • Actual procedure may differ from documented procedure
  • Quantitative data on volumes or timing is needed
  • Interviewing is insufficient for the task
  • Verifying information from other techniques
Hawthorne Effect: Staff change behavior when watched → less accurate data.

Data Storage & Program Design

Exam Q7 · 6 Marks · Flat file advantages · Top-down strategy

Flat File Storage — Two Advantages Must Memorize

Flat File Storage: Data stored in simple independent 2D table files managed directly by the OS — no DBMS needed.
  1. Simplicity — Easy to create and manage; no DBMS or DBA required
  2. Low cost — No DBMS license; uses the OS directly
  3. Fast sequential access — Efficient when hit rate is very high
  4. Application autonomy — Each app controls its own data independently

Top-Down Strategy in Modern Structured Program Design Understand

Top-Down Strategy: Design begins at the highest-level (main) module and progressively decomposes downward into sub-modules until all functions are defined.
  1. Identify the main module (e.g., "Process Customer Order")
  2. Break into major sub-modules (Receive Order, Validate, Send to Cashier, Print)
  3. Continue decomposing until each module performs exactly one task
  4. Draw a Structure Chart showing hierarchy and data passed between modules
  5. Implement top-down using stub modules for lower levels — critical interfaces tested first
HIGH cohesion (focused on one task) + LOOSE coupling (minimal inter-module dependency) = good program design.

Question 1 — Fundamentals of SA&D9 Marks

A — Define "Information Technology"

Information Technology (IT) is the use of hardware, software, services, and supporting infrastructure to manage and deliver information — including voice, data, and video — within an organization.

B — Two IS components; explain one in ≤ 3 lines

Two components: (1) People, (2) Technology.

Technology: Technology refers to the hardware, software, and communication networks that support the IS. In TRG, this includes the computers and printers in each branch restaurant, the servers at the ISC in HQ, and the communication network linking them together.

C — Two analyst/designer roles; explain one in ≤ 3 lines

Two roles: (1) System Analyst, (2) Systems Architect.

Systems Architect: Responsible for designing the overall technical structure of the IS — deciding which hardware, software, and networking technologies to use, how they relate to each other, and how processing is distributed. For TRG, the architect would design the client-server setup linking branch computers to ISC servers.
Question 2 — SDLC, Analysis Approaches & Info. Req. Analysis12 Marks

A — Define SDLC and state its abbreviation

The System Development Life Cycle (SDLC) is a logical, structured process by which analysts, engineers, programmers, and end users build information systems to solve business problems. It serves as a project management tool to plan, execute, and control the development process from start to finish.

B — Three phases of the SDLC

1. Planning — Identify the project; perform feasibility analysis; obtain steering committee approval.
2. Analysis — Investigate the current system; gather requirements; define the new system's needs. (Sub-phases: Survey, Detailed Study, Definition.)
3. Design & Prototyping — Design architecture, user interfaces, input/output, data storage, and programs.

C — Define business requirements; state two types

Business Requirements are the overall goals and capabilities an IS must fulfill to contribute to organizational success. They describe what the business needs, independent of implementation.

1. Functional Requirements — Specific behaviors the system must perform (e.g., "Register a customer order," "Send validated request to cashier").
2. Non-Functional Requirements — Qualities the system must have (e.g., "Response under 2 seconds," "Accessible from all 24 branches").

D — Two Analysis Strategies; explain one in ≤ 3 lines

Two strategies: (1) Process-Centered Analysis, (2) Data-Centered Analysis.

Process-Centered Analysis: This strategy focuses on identifying and modeling what processes the system must perform and how data flows between them. The primary tool is the Data Flow Diagram (DFD). Suitable when business processes are complex — as is the case for TRG's multi-department sales workflow.
Question 3 — System Analysis Approaches (Planning & Analysis Phases)6 Marks

A — Two Planning Phase sub-phases; explain one

Two sub-phases: (1) Project Initiation, (2) Project Management.

Project Initiation: The project sponsor identifies a business need and formally requests a new system by preparing a System Request. This document describes the business need, expected value, requirements, and constraints. The steering committee reviews it to decide whether to approve the project.

B — Two Analysis Phase sub-phases; explain one

Two sub-phases: (1) Survey Phase, (2) Detailed Study Phase.

Detailed Study Phase: Analysts perform a thorough investigation of the existing system. They model the current system using DFDs, analyze all business processes, identify problems and opportunities, define improvement objectives, and present findings to management. For TRG, this involves studying how orders are received, validated, and passed between departments across all 24 branches.

A — Conditions to apply "Observation"

1. The actual procedure may differ from the documented procedure.
2. Quantitative data on volumes or timing is needed (e.g., orders per hour in a branch).
3. Interviewing is insufficient — work involves hands-on tasks better seen than described.

• Questionnaires — Distribute to Sales staff in all 24 branches simultaneously — geographically dispersed, large number, efficient.

Question 5 — System Design: Building the System Technical Architecture6 Marks

A — Define "System Technical Architecture" in ≤ 3 lines

System Technical Architecture (STA) is the fundamental functional organization of a system — defining its components, their relationships, and the principles guiding its design. It specifies the technologies used for presentation, application logic, data access, and data storage, and describes how these functions are distributed across hardware, software, and communication networks.

B — Design the Technical Architecture for TRG Group Restaurants

The proposed architecture is a 3-Tier Client-Server architecture:

Tier 1 — Client (each branch): One computer (presentation & input) + one printer per restaurant. Used by Sales Dept. staff.
Tier 2 — Application Server (ISC, HQ): Hosts all business logic — receiving, validating, routing orders. Ensures centralized control across all 24 branches.
Tier 3 — Database Server (ISC, HQ): Stores all order data, customer records, and archives for all branches.
Communication: Branch computers connect to ISC servers via WAN, enabling orders from any branch.

Question 7 — Data Storage and Program Design6 Marks

A — Two flat file storage advantages; explain one in ≤ 3 lines

Two advantages: (1) Simplicity, (2) Low cost.

Simplicity: Flat files are simple 2D tables managed directly by the OS without requiring any DBMS software or a dedicated DBA. This makes them quick to set up and easy to understand for small, standalone applications where data relationships are not complex.

B — How to follow Top-Down strategy in Modern Structured Program Design

1. Define the main module (e.g., "Process Customer Order in ASDS").
2. Decompose into major sub-modules (Receive Order, Validate Order, Send to Cashier, Print Receipt).
3. Continue decomposing until each module performs exactly one simple, well-defined task (high cohesion).
4. Draw a Structure Chart showing module hierarchy and data passed between parent and child modules.
5. Implement top-down — code highest-level modules first; use stub (placeholder) modules for lower levels so critical interfaces are verified early.
