HTTP & The State Problem

Why cookies and sessions exist — the stateless nature of HTTP

Key Concepts

  • Stateless HTTP: the server does NOT remember the client between requests
  • Cookies: small text files stored on the client computer for tracking
  • Sessions: relevant data stored server-side, identified by a unique session ID
  • Use cases: shopping carts, "Remember Me" login, user preferences
  • PHP support: transparently supports HTTP cookies via built-in superglobals

  • Condition 1: Domain must match the cookie's domain attribute
  • Condition 2: Path must match the cookie's path attribute
  • Condition 3: Cookie must NOT be expired
  • Condition 4: Security rules (Secure, HttpOnly, SameSite) must allow
⚠ Critical Exam Point
  • HTTP is stateless by design — each request is completely independent
  • Cookies and sessions are application-level workarounds, NOT part of core HTTP
  • Cookie is sent in the HTTP header, NOT as a file attachment
ℹ HTTP Response Header Anatomy

When a PHP script calls setcookie(), it adds a Set-Cookie header to the HTTP response. The browser then stores this and automatically includes it in all subsequent requests matching the domain/path.

Setting Cookies (expires in 1 hour) 
<?php
setcookie("name", "John Watkin", time()+3600, "/");
setcookie("age",  "36",         time()+3600, "/");

// Loop all cookies
foreach($_COOKIE as $key => $value) {
    echo "{$key} = {$value} <br>";
}
?>
Accessing a specific cookie 
<?php
echo $_COOKIE["name"]; // John Watkin
echo $_COOKIE["age"];  // 36
?>
Checking if cookie exists with isset() 
<?php
if(isset($_COOKIE["name"])) {
    echo "Welcome " . $_COOKIE["name"];
} else {
    echo "Sorry... Not recognized";
}
?>
Deleting a cookie (set expiry in the past) 
<?php
// time()-60 = 60 seconds in the past
setcookie("name", "", time()-60, "/");
setcookie("age",  "", time()-60, "/");
?>
⚠ Critical Notes
  • setcookie() MUST be called before any HTML output including <html> tag
  • Each cookie requires a separate setcookie() call — no batch setting
  • To DELETE a cookie, set its expiry to a past timestamp (time()-60)
  • Cookie value is sent as plain text — never store passwords or sensitive data

Sessions In Depth

Server-side state management — starting, reading, and destroying PHP sessions

Key Concepts

  • session_start(): must be called before HTML — creates or resumes a session
  • $_SESSION[]: superglobal array — stores all session variables
  • SESSID Cookie: browser stores only a session ID — actual data stays on server
  • Auto-delete: session values are wiped when browser closes
  • Persistence: for permanent storage, save session data to a database
  • Global variable: a session is a global variable stored on the server

When to Use Sessions

  • Sensitive data: store user ID securely — tamper-proof on server
  • Multi-page data: pass values between different PHP pages
  • Cookie-free fallback: works on browsers with cookies disabled
  • Global variables: safer than passing in URL query strings
  • Large data: shopping cart data exceeding the 4KB cookie limit
Starting a session & storing variables 
<?php
session_start(); // MUST be before HTML
?>
<html><body>
<?php
$_SESSION["FName"] = "ABC";
$_SESSION["Lname"] = "XYZ";
echo "First: " . $_SESSION["FName"];
echo "Last: "  . $_SESSION["LName"];
?>
</body></html>
Page visit counter with sessions 
<?php
session_start();
if(isset($_SESSION['page_count'])) {
    $_SESSION['page_count'] += 1;
} else {
    $_SESSION['page_count'] = 1;
}
echo 'Visitor #' . $_SESSION['page_count'];
?>
Destroying a specific session variable 
<?php
session_start();
// Remove only one variable
unset($_SESSION['product_item']);
// Other session data remains intact
?>
Destroying the ENTIRE session 
<?php
session_start();
session_unset();   // clear all variables
session_destroy(); // destroy session + cookie
header("Location: login.php");
exit();
?>
⚠ session_destroy() vs unset() — Know the Difference
  • session_destroy() — removes ALL session data including the SESSID cookie
  • unset($_SESSION['key']) — removes ONE variable; other session data stays intact
  • Always call session_unset() before session_destroy() for a clean logout
✓ Session Flow Summary

session_start() → set $_SESSION variables → access on other pages (also call session_start first) → logout with session_unset() + session_destroy()

Cookies vs Sessions — Full Comparison

Side-by-side comparison of all key attributes — a high-priority exam topic

Comparison Table

Attribute 🍪 Cookies 🔐 Sessions
Storage Location Client (Browser) Server
Data Size Limit ~4KB max No strict limit
Security Lower — user can read/edit Higher — data stays on server
Expiry Custom expiry date or browser close Browser close (unless DB-persisted)
PHP Superglobal $_COOKIE $_SESSION
Start/Set Function setcookie() session_start()
Browser Dependency Requires browser to accept cookies Works even if cookies blocked*
Data Type String only Any PHP type (arrays, objects)
Server Load None — client holds data Some — server stores data
Ideal Use Case Remember Me, preferences, tracking Login auth, shopping cart, user state

* Sessions still use a cookie (SESSID) to identify the user unless URL rewriting is configured

⚠ Cookies — Risks
  • Visible and editable by the user via browser DevTools
  • Can be stolen via XSS if HttpOnly not set
  • Subject to CSRF attacks if SameSite not configured
  • Limited to 4KB — cannot store complex data structures
✓ Sessions — Advantages
  • Data never leaves the server — much more secure
  • Can store arrays, objects, complex state
  • No size restriction on stored data
  • Automatically ties to browser lifecycle
ℹ How Sessions Use Cookies Internally

When session_start() is called, PHP creates a unique session ID and stores it as a cookie named PHPSESSID on the client. On every subsequent request, the browser sends this cookie, and PHP uses the ID to locate and load the session data from the server's file system (or database).

login.php — Show form, handle errors 
<?php
session_start();
// Redirect if already logged in
if (isset($_SESSION['username'])) {
    header("Location: dashboard.php");
    exit();
}
?>
<!-- HTML form posts to process.php -->
<?php
// Show error if one was set
if (isset($_SESSION['error'])) {
    echo $_SESSION['error'];
    unset($_SESSION['error']);
}
?>
process.php — Validate credentials 
<?php
session_start();
$valid_user = "admin";
$valid_pass = "1234";
$u = $_POST['username'];
$p = $_POST['password'];

if ($u === $valid_user && $p === $valid_pass) {
    $_SESSION['username'] = $u;
    header("Location: dashboard.php");
} else {
    $_SESSION['error'] = "Invalid credentials!";
    header("Location: login.php");
}
exit();
?>
dashboard.php — Protected page 
<?php
session_start();
// Guard clause — redirect if not logged in
if (!isset($_SESSION['username'])) {
    header("Location: login.php");
    exit();
}
?>
<h2>Welcome,
<?php echo $_SESSION['username']; ?>
</h2>
<a href="logout.php">Logout</a>
logout.php — Full session teardown 
<?php
session_start();
// Step 1: Clear all session variables
session_unset();
// Step 2: Destroy the session itself
session_destroy();
// Step 3: Redirect to login
header("Location: login.php");
exit();
?>
⚠ Security Notes on This Pattern
  • Always use session_unset() before session_destroy() for a complete logout
  • Add exit() immediately after every header("Location: ...") to stop execution
  • In production, hash passwords with password_hash() — never compare plain text
  • The "guard clause" pattern in dashboard.php must appear at the very top of every protected page
