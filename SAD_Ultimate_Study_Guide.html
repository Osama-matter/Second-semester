1. Fundamentals — Information Systems & Roles 🔥 Exam Q1

▼

🔥 Exam Definition — ALWAYS ASKED

Information Technology (IT): The use of computer hardware, software, and networks to collect, process, store, and distribute information to support decision-making and organizational activities.
Information System (IS): A set of interrelated components that collect, process, store, and distribute information to support decision making and control in an organization.
System Analysis: The process of studying and understanding an existing system to identify its problems and requirements for improvement or replacement.
System Design: The process of planning a new system or modifying an existing one to satisfy the specified requirements.

Components of an Information System FREQUENT

H S P D N
Hardware · Software · People · Data · Network/Procedures
  • Hardware: Physical devices — computers, printers, servers
  • Software: Programs that run on hardware (OS, applications)
  • People: Users, analysts, operators, managers
  • Data: Raw facts that are processed into information
  • Network/Procedures: Rules and connections between components

Roles of System Analyst ⭐ FREQUENT

CIPDAM
Communicator · Investigator · Problem Solver · Designer · Architect · Manager
  • Communicator between users and technical team
  • Investigator — collects facts about current system
  • Problem identifier — detects shortcomings
  • Designer — designs new system components
  • Architect — builds system structure
  • Project coordinator — manages team and schedule

⭐ "Project Sponsor" — Asked in 2024!

The Project Sponsor is the senior person (usually top management) who authorizes and funds the project. They define the business goals and are the champion of the project within the organization.

⚡ 30-Second Revision

IT = use of tech for info · IS = 5 components (HSPDN) · Analyst = bridge between users & tech · Sponsor = the boss who funds it
🔥 Exam Q (2024, 2025) — Define + Explain
A. Define "Information Technology". B. State two components of an IS and explain one. C. State two roles of the system analyst and explain one.
✅ Model: Define IT clearly (1 line). Pick 2 from HSPDN (Hardware, Software). For roles: pick Investigator + Designer. Explain in 2–3 lines each. Never write more than 3 lines per explanation — examiners want concise!

2. SDLC — System Development Life Cycle 🔥 Exam Q2

▼

🔥 ALWAYS ASKED — Define SDLC

SDLC (System Development Life Cycle): A structured process that defines the phases and activities involved in developing, maintaining, and replacing information systems. It provides a framework for planning and controlling the development of an information system.
System Development Methodology (SDM): The specific approach or set of procedures used to implement the SDLC. The SDM specifies how the work is done within each phase. SDLC = "what phases", SDM = "how to do them".

The 3 Main Phases of SDLC 🔥 State 3 phases

A D I
Analysis Phase · Design Phase · Implementation Phase

📋 Analysis Phase

Understand the current system, define problems, gather requirements

  • Survey sub-phase
  • Detailed Study sub-phase
  • Definition sub-phase

🎨 Design Phase

Plan the new system solution

  • Logical Design (what)
  • Physical Design (how)

⚙️ Implementation Phase

Build, test, and deploy

  • Construction
  • Testing
  • Deployment/Maintenance

Business Requirements ⭐ Define + 2 Types

Business Requirements: The conditions or capabilities that a system must fulfill to satisfy the needs of the users and the business objectives.

Functional Requirements

What the system MUST DO — specific functions and behaviors

Example: "The system shall allow a customer to place an order"

Non-Functional Requirements

How the system performs — quality attributes

Example: "Response time < 2 seconds", "99.9% uptime"

Analysis Strategies 🔥 Mention 2 + Explain 1

Top-Down Analysis

Start from the WHOLE system overview, then break it into smaller parts. Like reading a book title before chapters.

Used when: redesigning a completely new system

Bottom-Up Analysis

Start from individual DETAILS and work up to the big picture. Like assembling puzzle pieces.

Used when: improving an existing system

Project Payments Milestones ⭐ Asked 2024!

Project Payments Milestones: Predefined checkpoints in the project timeline at which payments are made to the development team based on the successful completion of specific deliverables or phases.

5 Types of Requirements (System Design Exam) 🔥 Asked 2024

PFIOP
Performance · Functional · Information · Operational · Protection(Security)
  • Functional: What the system does (functions/features)
  • Performance: Speed, capacity, response time
  • Information: Data storage and retrieval needs
  • Operational: How system fits the work environment
  • Protection (Security): Access control, data security

⚡ 30-Second Revision

SDLC = 3 phases (ADI) · SDM = how to do SDLC · Requirements = Functional + Non-Functional · Strategies = Top-Down & Bottom-Up

3. System Analysis Phases in Detail 🔥 Exam Q3/Q5

▼

The Analysis Phase — 3 Sub-Phases

S D D
Survey Phase · Detailed Study Phase · Definition Phase

🔥 Survey Phase Activities (Sub-phases)

P P D
Plan the Project · Prepare the Plan · Define Project Scope
  • Plan the Project: Assign the project sponsor, define objectives, set rough schedule and budget, determine feasibility
  • Assign a Planner: The project manager assigns a planner because the survey phase creates plans that require special planning expertise
  • Prepare the Plan: Create detailed project plan with time, cost, resources

⭐ Why assign a Planner in Survey Phase?

Because the survey phase involves creating a project plan (schedule, budget, resources). Planning requires specialized skills in estimation and project management — skills the analyst may not have. So a dedicated planner is assigned.

🔥 Detailed Study Phase Activities

  • Collect Data: Apply fact-finding techniques (interviews, questionnaires, observation, record inspection)
  • Analyze Current System: Understand existing processes, identify problems
  • Prototyping: Build working model to discover and verify user requirements
  • Define Requirements: Document what the new system must do

📦 Prototyping — 3 Characteristics (Asked 2024!)

  • Built quickly with minimum effort and cost
  • Provides a working model that users can interact with
  • Helps discover and verify user requirements early
  • May be throwaway (discarded after requirements confirmed) or evolutionary (becomes the final system)

Definition Phase

  • Assign a modeler — because the definition phase produces formal models (DFDs, ERDs) that require modeling expertise
  • Produce the System Requirements Specification (SRS)
  • Prioritize requirements — because not all requirements are equally important; management must decide which to include based on cost and value
🔥 Exam Pattern 2024 (Q5 — 30 Marks!)
State objectives of "Plan the Project" / Why assign planner in survey phase? / Prototyping characteristics / Why prioritize requirements in definition phase? / Why assign modeler?
✅ These are all short-answer "explain why" questions. Write 2–3 lines each. Be very concise. Don't pad with extra words.

⚡ 30-Second Revision

Survey = Plan project · Detailed Study = collect data + prototype · Definition = SRS + model · Planner assigned = planning expertise · Modeler assigned = DFD/ERD expertise

4. Fact-Finding Techniques 🔥 Exam Q3/Q4

▼

The 5 Fact-Finding Techniques

IQORI
Interview · Questionnaire · Observation · Record Inspection · Internet/Research

🔥 Observation — Conditions to Use

Use observation when:

  • Users cannot clearly describe their work processes
  • You need to verify what was said in interviews
  • The process is complex and visual understanding is needed
  • Existing documentation is inadequate or missing

🔥 Questionnaire — Conditions to Use

Use questionnaire when:

  • You need data from many people spread across locations
  • Respondents are geographically distant
  • You need statistical/quantitative data
  • Time and cost are limited

⭐ The TRG/RRG Restaurant Case — Repeated Every Year!

The pattern: "State the fact-finding techniques needed to collect data from HQ and individual restaurants"

For HQ (Head Quarter):
  • Interview — with HQ managers and department heads
  • Record Inspection — examine HQ documents, reports, archives
  • Observation — observe HQ workflows directly
For Individual Restaurants (branches):
  • Questionnaire — because there are MANY branches (24–64 branches) spread over different governorates; can't visit all
  • Interview — with branch sales managers (a sample)

🔑 KEY REASON: Questionnaire for branches because there are too many (24/64) spread geographically — you can't interview them all!

🔥 Exact Exam Question — 2024, 2025
"State the fact-finding techniques used to prepare a plan to collect data and investigate the problems of the TRG Group restaurants (HQ and individual restaurants)"
✅ Answer: For HQ → Interview + Record Inspection + Observation. For branches → Questionnaire (because 24+ branches spread across Egypt). Give reason WHY for each technique.
TechniqueBest ForAdvantageDisadvantage
InterviewFew key people, detailed infoRich detail, clarification possibleTime consuming, costly
QuestionnaireMany people, spread outFast, cheap, large sampleLow response rate, no clarification
ObservationComplex processesAccurate, first-handHawthorne effect, time-consuming
Record InspectionDocuments, reports, formsObjective data, historicalMay be outdated or incomplete

⚡ 30-Second Revision

Observation = when users can't explain their work · Questionnaire = many distant people · HQ = Interview+Inspection+Observation · Branches = Questionnaire (too many!)

5. System Modeling — DFD (Data Flow Diagrams) 🔥 HIGHEST MARKS

▼

🔥 What is System Modeling?

System Modeling: Developing and presenting an abstract blueprint of a system using graphical notation to help analysts understand and communicate system structure and behavior.

Purpose of modeling techniques (2 main ones):

  • Understanding: Help the analyst understand complex systems
  • Communication: Help communicate system design to users and team

DFD Components — 4 Symbols

EPDS
External Entity (rectangle) · Process (rounded rectangle/circle) · Data Store (open rectangle) · Data Flow (arrow)

□ External Entity

A person, organization, or system OUTSIDE the system. Source or destination of data.

Example: Customer, Manager, Government

⬭ Process

An activity that transforms input data into output data. Named as action verbs.

Example: "Validate Order", "Generate Report"

▭ Data Store

Where data is stored (file, database, archive). Shown as open-ended rectangle.

Example: Archive, Customer File, Orders DB

→ Data Flow

Movement of data between components. Named to describe what data is flowing.

Example: "Requested Order", "Validated Request"

DFD Levels

🔥 3 Levels — Must Know Cold!

Context
Level 0 (sometimes called)
  • ONE process only
  • Shows ALL external entities
  • Shows ALL data flows in/out
  • No data stores
  • System = single bubble
Level-0
(Main processes)
  • Expands the context process
  • Shows MAIN processes (departments)
  • Shows data stores
  • Same external entities
  • Must BALANCE with context
Level-1
(Sub-processes)
  • Expands ONE Level-0 process
  • Shows sub-processes inside it
  • Must BALANCE with Level-0
  • More detail

🔥 The Balancing Rule — VERY IMPORTANT

⚠️ BALANCING RULE: The inputs and outputs of a process at one level MUST MATCH the inputs and outputs of the same process when expanded to the next level.

If Level-0 "Sales Dept" receives "Requested Order" from Customer → then when you expand Sales Dept to Level-1, it must STILL receive "Requested Order" from Customer.

🔥 Restaurant Case — DFD Step by Step (TRG / RRG)

STEP 1: Context Diagram
CONTEXT DIAGRAM — TRG/RRG Restaurant System ════════════════════════════════════════════════ [Customer] ──Requested Order──→ [ RESTAURANT ] ──Requested Items──→ [Customer] [ MANAGEMENT ] [ SYSTEM ] ──Requested Receipt──→ [Customer] ↑ (ONE process = whole system) Note: • Only ONE big process bubble • External entities: Customer (appears twice - same entity) • Show ALL flows in and out • NO data stores at context level
STEP 2: Level-0 DFD
LEVEL-0 DFD — TRG Restaurant ════════════════════════════════════════════════ [Customer]──Requested Order──→ [1. Sales Dept] │ Validated Request ↓ [2. Cashier Dept]──Validated Request──→ [3. Meal Production] ↑ │ Requested Receipt Prepared Request ↓ [4. Service Delivery]──→ [Customer] (items+receipt) Data Stores: [Archive] connected to Sales, Cashier, Meal Production External Entities: Customer (source and destination)
STEP 3: Level-1 DFD (Sales Department)
LEVEL-1 DFD — Sales Department Detail ════════════════════════════════════════════════ [Customer] ──Requested Order──→ [1.1 Register Order] │ Requested Order ↓ [1.2 Validate Order] │ Validated Request ↓ (to Cashier) / [1.3 Print Request] / [Archive] Note: All inputs/outputs MUST BALANCE with Level-0 Sales Dept • Input from Customer: Requested Order ✓ (same as Level-0) • Output to Cashier: Validated Request ✓ (same as Level-0)

Common DFD Errors

BMSEU
Black Hole · Miracle · Self-loop · Entity-to-Entity · Unbalanced

❌ Black Hole

A process has INPUT but NO OUTPUT. Data goes in and disappears.

❌ Miracle

A process has OUTPUT but NO INPUT. Data appears from nowhere.

❌ Self-Loop

A data flow goes FROM a process BACK TO the same process.

❌ Entity-to-Entity

A data flow goes directly between two external entities (no process involved).

✅ How to Answer DFD Questions in the Exam

  1. Read the case study carefully — identify external entities (people/orgs interacting with the system)
  2. Identify all data flows (what documents/data are exchanged?)
  3. For Context: Draw ONE bubble, put entities on sides, draw arrows with data names
  4. For Level-0: Each department = one process bubble. Connect with same flows as context (BALANCE!)
  5. For Level-1: Pick ONE department from Level-0, break it into sub-steps
  6. Always LABEL every arrow with the data name
  7. Check balancing before finishing

⚡ 30-Second Revision

Context = 1 process · Level-0 = departments · Level-1 = sub-steps of 1 dept · Balance = same in/out · Errors = BMSEU · Restaurant: Customer → Sales → Cashier → Meal Prod → Delivery → Customer

6. System Technical Architecture (STA) 🔥 Exam Q5/Q6

▼

🔥 Define STA — Always Asked!

System Technical Architecture (STA): A description of the hardware, software, and network components needed to implement an information system, including the arrangement of computers, servers, and communication infrastructure.

3 System Acquisition Strategies 🔥 Asked in 2024 design exam — 15 marks!

BCM
Buy (COTS) · Custom Build · Modify/ERP

1. Buy (COTS)

Commercial Off-The-Shelf software

  • Ready-made product
  • Fast deployment
  • Lower cost initially
  • Less customization

2. Custom Build

Build the system from scratch internally or outsource

  • 100% fits requirements
  • Most expensive
  • Takes longest time
  • Full control

3. Modify Existing

Buy a system and customize it (ERP approach)

  • Balance of cost & fit
  • Medium time
  • Partial customization

Architecture Types 🔥

🔥 Client-Server vs Server-Based — Most Repeated!

AspectClient-ServerServer-Based
ProcessingSplit between client & serverALL on server
Client roleDoes some processingJust displays results (thin)
CostHigher (powerful clients needed)Lower (cheap thin clients)
ScalabilityHarder to scaleEasy to scale
ExampleThick client appsWeb browser apps

Thick Client vs Thin Client

Thick (Fat) Client

  • Powerful local machine
  • Stores data & apps locally
  • Works offline
  • Higher hardware cost
  • Harder to maintain/update

Thin Client

  • Minimal local processing
  • Everything on server
  • Requires network always
  • Cheaper hardware
  • Easy central management

Tiers Architecture

2-Tier

Client ↔ Server

Direct client-server connection. Simple, fast, limited scalability.

3-Tier

Client ↔ App Server ↔ DB Server

Presentation + Logic + Data separated. More scalable.

N-Tier

Multiple specialized servers

Enterprise scale. Each layer is independent.

⭐ How to Design STA (Drawing Question)

For the Restaurant/Clinic case, the STA always follows this pattern:

ARCHITECTURE DESIGN — TRG Restaurant (Client-Server) ════════════════════════════════════════════════════ [HQ/ISC - Alexandria] ┌─────────────────────────────┐ │ Server 1 (Main) │ │ Server 2 (Backup) │ └─────────────┬───────────────┘ │ Communication Network (Internet/WAN) ┌──────────┼──────────┐ ↓ ↓ ↓ [Branch 1] [Branch 2] ... [Branch 24] ┌────────┐ ┌────────┐ │ PC │ │ PC │ │ Printer│ │ Printer│ └────────┘ └────────┘ KEY COUNTS TO JUSTIFY: - 1 PC + 1 Printer per branch = 24 PCs + 24 Printers - 2+ Servers at HQ (for reliability/load) - Network: WAN connecting all branches to HQ

🔑 Always justify WHY: "We need 24 PCs because there are 24 branches, one per branch to handle local input"

Single Seat — Explained

In client-server architecture, "Single Seat" happens when all users try to use the same server resource and only one can proceed at a time — creating a bottleneck. This is why multiple servers or load balancing is needed.

⚡ 30-Second Revision

STA = hardware+software+network plan · 3 strategies = Buy/Build/Modify · Client-Server = split processing · Server-Based = all on server · 2-tier=C↔S, 3-tier=C↔App↔DB · Restaurant STA = PC+Printer per branch + servers at HQ

7. Input, Output & Interface Design 🔥 Exam Q6/Q7

▼

Making Input Easy & Efficient 🔥 ALWAYS ASKED

MCADVD
Minimize keystrokes · Code lookup · Auto-fill defaults · Drop-downs · Validation · Design for flow

🔥 How to Make Input Easy — Exam Answer

  • Minimize keystrokes: Use codes, dropdowns, checkboxes instead of typing
  • Automatic entry: System auto-fills date, user ID, defaults
  • Consistency: Use same layout and tab order as paper forms
  • Validation: Check data as entered (range checks, format checks)
  • Clear labels: Every field clearly labeled
  • Help messages: Guide user with hints and error messages

Validation Controls (Internal Controls for Accuracy)

RFTCP
Range check · Format check · Type check · Completeness check · Presence check
  • Range Check: Value within min/max (age: 0-120)
  • Format Check: Correct format (date: DD/MM/YYYY)
  • Type Check: Correct data type (number field = numbers only)
  • Completeness Check: Required fields not left empty
  • Reasonableness Check: Is the value logical? (salary: 500 EGP seems too low)
  • Uniqueness Check: No duplicate IDs

Output Design Objectives ⭐ State 3

RACTU
Relevance · Accuracy · Completeness · Timeliness · Usability
  • Relevance: Only needed information included
  • Accuracy: Output is correct and reliable
  • Completeness: All required data is present
  • Timeliness: Output available when needed
  • Usability: Easy to read and understand

Output Methods ⭐

  • Printed (Hard Copy): Physical paper output — receipts, reports
  • Screen (Soft Copy): Displayed on monitor — queries, dashboards
  • Electronic: Email, PDF, exported files

Interface Design — Standard Options & Consistency

⭐ "Standard Options and Consistency" — Asked in 2024!

This guideline means: Use the SAME design patterns everywhere in the system so users don't have to relearn.

Why important: Consistency reduces errors, speeds up learning, and makes the system easier to use.

2 ways to apply:

  • Use same menu structure, button labels, and colors throughout the system
  • Follow operating system standards (e.g., Ctrl+S = Save always works)

4 Ways to Keep Screens Consistent

LCNF
Layout · Colors · Naming · Feedback
  • Same screen layout (header, footer, menu position)
  • Same color scheme for same types of elements
  • Same terminology for same concepts
  • Same feedback messages for same actions

Easy input = MCADVD · Validation = RFTCP · Output objectives = RACTU · Interface = consistency + standard options · Input screen = auto fields + validation + clear labels

8. Database Design 🔥 Exam Q8/Q9

▼

2 Approaches to Data Storage

Flat File (Conventional File) Approach

Each application stores its OWN separate files.

Advantages:
  • Simple to implement
  • No special software needed
  • Fast for simple, single-app use
Disadvantages:
DIRSU
Duplication · Inconsistency · Rigidity · Security weak · Unshared data
  • Data duplication — same data stored multiple times
  • Data inconsistency — different versions of same fact
  • Rigid structure — hard to change
  • Poor security control
  • Data not shared between apps

Database Approach

Centralized database shared by all applications via DBMS.

Advantages:
RSICB
Redundancy reduced · Shared data · Integrity · Consistency · Better security
  • Reduced redundancy — data stored once
  • Shared across all applications
  • Data integrity enforced
  • Consistent data everywhere
  • Better security through access control
Disadvantages:
  • Complex and expensive to set up
  • Single point of failure risk
  • Requires trained DBA

🔥 DBA — Roles AND Harm (Asked 2024!)

While DBA plays a crucial role, they can also HARM the organization.

DBA Roles (Positive):
  • Design and create the database structure
  • Maintain database performance
  • Ensure data security and access control
  • Backup and recovery management
  • Train users on database use
DBA Harm (Negative):
  • Can access ALL data — privacy risk
  • Can modify or delete critical data
  • Can grant unauthorized access to others
  • If DBA leaves, organization may lose expertise

Database Design Objectives ⭐ State 3 + Explain 2

RIASC
Redundancy reduction · Integrity · Accessibility · Security · Consistency

Entity Types in Database Design ⭐ Asked 2019

  • Independent Entity: Exists on its own (Customer, Product)
  • Dependent Entity: Existence depends on another entity (Order Item depends on Order)
  • Associative Entity: Represents a relationship between two entities (Enrollment = Student + Course)

File Organization Methods

Sequential

Records stored in order. Good for batch processing. Slow for random access.

Direct/Random

Access any record directly by address. Fast retrieval. Needs index.

Indexed Sequential (ISAM)

Combines both — has index for fast access, records in sequence.

⚡ 30-Second Revision

Flat file disadvantages = DIRSU · Database advantages = RSICB · DBA roles = design+security+backup · DBA harm = can see/change all data · Entity types = Independent/Dependent/Associative

9. Program Design — Modular & Structured 🔥 Exam Q9

▼

🔥 Modular Programming — Define + 2 Advantages

Modular Programming: A design technique that breaks a large complex program into smaller, independent, manageable units called modules, each performing a specific function.
Advantages:
RETMD
Reusability · Easy maintenance · Team work · Manageability · Debugging ease
  • Easy to maintain — fix one module without touching others
  • Reusable — modules can be used in other programs
  • Enables teamwork — different developers work on different modules
  • Easier to test and debug — test each module independently
  • Reduces complexity — divide and conquer approach

2 Properties of Good Module Design 🔥 Asked 2024!

🔥 High Cohesion

Cohesion: The degree to which elements INSIDE a module are related to each other and work toward a single purpose.

High cohesion = GOOD → module does ONE thing well

Think: a specialist doctor who only does heart surgery

🔥 Low Coupling

Coupling: The degree of interdependence BETWEEN modules — how much one module depends on another.

Low coupling = GOOD → modules are independent

Think: a standalone app that doesn't need other apps to work

⚠️ Memory Trick: "HIGH Cohesion, LOW Coupling" = HCLC = "He Codes Like Champions" 🏆

Top-Down vs Bottom-Up Strategies 🔥

AspectTop-DownBottom-Up
Start fromMain module (overview)Detailed sub-modules
DirectionGeneral → SpecificSpecific → General
TestingTest main logic firstTest details first
Use whenNew system designReusing existing components
AnalogyBuilding a house — roof plan firstBuilding with LEGO — pieces first

⭐ How to Follow Top-Down Strategy (Exam Answer)

In Modern Structured Program Design, to follow Top-Down:

  1. Start by designing the main control module which calls all others
  2. Break it into sub-modules at the next level
  3. Continue breaking down until modules are small and manageable
  4. Each lower module handles a single specific function
  5. Use structure charts to visualize the hierarchy

2 Decomposition Strategies (Asked 2019)

Functional Decomposition

Break system into functions/processes. Focus on WHAT the system does.

Object-Oriented Decomposition

Break system into objects. Focus on WHO does what (objects with attributes and methods).

⚡ 30-Second Revision

Modular = break into modules (RETMD) · Good design = High Cohesion + Low Coupling (HCLC) · Top-down = general→specific · Bottom-up = specific→general · Decomposition = Functional OR OO

🔥 ONE-LINE DEFINITIONS (Write these EXACTLY in exam)

IT: Use of computer hardware/software/networks to collect, process, store, and distribute information.
IS: Set of interrelated components (HSPDN) that process and distribute information to support decisions.
SDLC: Structured process of phases (ADI) for developing, maintaining, and replacing information systems.
SDM: Specific approach used to implement the SDLC (SDLC = what phases, SDM = how to do them).
STA: Description of hardware, software, and network components needed to implement a system.
Modular Programming: Breaking a large program into small, independent modules each doing one function.
Cohesion: Degree to which elements inside a module work toward a single purpose. (Want: HIGH)
Coupling: Degree of interdependence between modules. (Want: LOW)
Prototype: A working model built quickly to discover and verify user requirements.
Business Requirements: Conditions the system must fulfill to satisfy user and business needs.
Project Sponsor: Senior person who authorizes and funds the project.
System Modeling: Developing abstract blueprints of a system using graphical notation (DFDs, ERDs).

🎯 DFD Rules — Never Forget

Context = 1 process, no data stores
Level-0 = expand context process into departments
Level-1 = expand ONE Level-0 process into sub-steps
Balancing = in/out flows must match between levels
Label EVERY arrow with data name
Errors = Black Hole, Miracle, Self-loop, E-to-E, Unbalanced

📋 Common Exam Mistakes to AVOID

❌ Writing more than 3 lines for "explain in not more than 3 lines" questions — examiner deducts marks!
❌ Forgetting to BALANCE your DFD between levels
❌ Drawing a context diagram with multiple processes — context = ONE bubble only!
❌ Confusing SDM with SDLC (SDLC = phases; SDM = method to do them)
❌ Confusing Cohesion (internal) with Coupling (external between modules)
❌ Saying questionnaire for HQ — it's interview! Questionnaire = for 24+ distant branches
❌ Not labeling data flows on DFD arrows
