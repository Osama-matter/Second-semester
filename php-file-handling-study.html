01 / Foundation

The Golden Rule

Before touching any file, understand what can go wrong — permanently.

🚨 CRITICAL: File Operations Are Destructive & Irreversible

Unlike databases with transactions, filesystem operations have no undo. The wrong mode or function can silently delete data in milliseconds.

  • Opening a file with w truncates it to zero bytes — immediately, even before you write anything
  • Overwriting a file is permanent; there is no recycle bin on the server
  • Race conditions between concurrent requests can corrupt files
  • Writing outside the intended directory is a classic path traversal attack

The Three Commandments

① Always close what you open
Every fopen() must have a matching fclose(). Unclosed file handles leak resources, can prevent other processes from writing, and may corrupt data on some systems.
⚠ Resource leak without fclose()
② Check return values
fopen() returns false on failure. Never assume a file opened successfully. Always validate before reading or writing — a false handle will silently do nothing or throw warnings.
✓ if ($handle === false) die("...")
③ Choose the right mode
The difference between "r" and "w" is the difference between reading data and destroying it. Double-check your mode before opening production files.
r = safe read · w = destructive
④ Never trust user input for paths
User-supplied filenames can contain ../ traversal sequences. A request for ../../etc/passwd could expose or overwrite system files. Always sanitize and validate paths.
⚠ Path traversal vulnerability ✓ Use basename() + realpath()

Safe File Operation Template

example.php
// ✅ The correct pattern every time
$path = '/var/data/users.txt';

$handle = fopen($path, 'r'); // ① Choose mode carefully

if ($handle === false) {            // ② Always check success
    throw new RuntimeException("Cannot open file: $path");
}

try {
    while (($line = fgets($handle)) !== false) {
        processLine(trim($line));
    }
} finally {
    fclose($handle);               // ③ Always close — even on error
}
💡 Exam Tip: Examiners love to test whether you remember fclose(). In any question about file handling code, always check that handles are closed — especially in a finally block so cleanup happens even if an exception is thrown.
02 / Core Concept

File Opening Modes

The second argument to fopen() controls what you can do and what happens to existing content.

🚨 Danger Zone: The Destructive Modes

w and w+ erase the entire file the moment fopen() is called — even if you never write a single byte. If the file doesn't exist, w/w+ create it. x/x+ are safer: they fail if the file exists, preventing accidental overwrites.

The fopen() Mode Reference

Mode Full Name Read Write Pointer Start File Must Exist? Truncates? Use Case
r Read Only ✓ Yes ✗ No Beginning Yes — fails if missing No Safely reading a config or data file
r+ Read/Write ✓ Yes ✓ Yes Beginning Yes — fails if missing No Patching specific bytes in an existing file
w Write Only ✗ No ✓ Yes Beginning No — creates if missing ⚠ YES — immediate! Replacing a file with entirely new content
w+ Read/Write (trunc) ✓ Yes ✓ Yes Beginning No — creates if missing ⚠ YES — immediate! Write-then-verify workflows on new content
a Append Only ✗ No ✓ Yes End No — creates if missing No Log files, audit trails, growing datasets
a+ Read/Append ✓ Yes ✓ Yes End No — creates if missing No Appending to a file while also reading it
x Exclusive Write ✗ No ✓ Yes Beginning No — fails if exists! N/A Creating unique files safely (prevents overwrite)
x+ Exclusive R/W ✓ Yes ✓ Yes Beginning No — fails if exists! N/A Atomic file creation with immediate read-back

Binary Mode Modifier

Append b to any mode (e.g., "rb", "wb") to open in binary mode. On Windows this prevents newline translation (\r\n ↔ \n). On Linux/Mac it has no effect but is good practice for images, PDFs, and other binary files — it documents intent.
modes.php
// ─── Safe: read-only, file must already exist
$f = fopen('data.txt', 'r');

// ─── DANGER: truncates instantly, even before fwrite()
$f = fopen('data.txt', 'w');  // data.txt is now EMPTY

// ─── Safe append: never truncates
$f = fopen('log.txt', 'a');

// ─── Safest write: fails if file exists (no accidental overwrite)
$f = fopen('newfile.txt', 'x');

// ─── Binary mode for images/PDFs
$f = fopen('photo.jpg', 'rb');
03 / Functions

Reading Functions

Four functions — each with a distinct purpose, memory profile, and use case.

readfile()
readfile(string $filename): int|false
Reads a file and immediately sends it to the output buffer. Does not return the content — returns the byte count. Ideal for file downloads. No handle needed.
✓ Memory efficient Output direct to browser Cannot manipulate content
file_get_contents()
file_get_contents(string $filename): string|false
Reads an entire file into a string. Simple one-liner, no handle needed. Also works with URLs. Can use $offset and $length to read a portion.
⚠ Loads all to RAM ✓ Returns string Works with URLs
fread()
fread(resource $handle, int $length): string|false
Reads up to $length bytes from an open handle. Precise control — use filesize() to read everything, or loop for chunked processing of large files.
✓ Chunked reading Requires fopen() handle Binary safe
fgets()
fgets(resource $handle, int $length = ?): string|false
Reads one line at a time (up to newline or EOF). Returns false at end of file. Perfect for parsing CSVs, log files, or any line-oriented format.
✓ Line-by-line parsing ✓ Memory efficient Requires fopen() handle

Decision Guide: Which to Use?

Use readfile() when…

  • Serving a file download directly to the user
  • You don't need to process the content at all
  • Memory efficiency is critical (large files)
  • Example: Content-Disposition: attachment downloads

Use file_get_contents() when…

  • You need the whole file as a string to manipulate it
  • Doing search-and-replace on file content
  • Fetching a remote URL response
  • File is small enough to fit comfortably in RAM

Use fread() when…

  • Reading binary files (images, zip, video)
  • Streaming large files in chunks
  • Reading a fixed number of bytes at a specific offset
  • Working with binary protocols or structured data

Use fgets() when…

  • Processing a file line-by-line (CSV, logs)
  • File is too large to load entirely into memory
  • You want to stop reading partway through
  • Parsing structured text formats

Code Examples

readfile() — file download
$file = '/files/report.pdf';
header('Content-Type: application/pdf');
header('Content-Disposition: attachment; filename="report.pdf"');
header('Content-Length: ' . filesize($file));
readfile($file); // Streams directly to browser — returns byte count
file_get_contents() — read & manipulate
$content = file_get_contents('template.html');
$output  = str_replace('{{NAME}}', 'Alice', $content);
echo $output;
fread() — chunked binary read
$handle = fopen('largefile.bin', 'rb');
while (!feof($handle)) {
    $chunk = fread($handle, 8192); // 8 KB per read
    processChunk($chunk);
}
fclose($handle);

// Or read entire small file at once:
$data = fread($handle, filesize('config.bin'));
fgets() — line-by-line CSV parsing
$handle = fopen('users.csv', 'r');
while (($line = fgets($handle)) !== false) {
    $row = str_getcsv(trim($line));
    insertUser($row[0], $row[1]); // name, email
}
fclose($handle);
💡 Key Distinction: readfile() outputs to the browser and returns a count. file_get_contents() returns the content as a string for you to use in PHP. fread() and fgets() both require an open handle from fopen().
04 / Functions

Writing Functions

Two approaches: the convenience wrapper and the low-level handle method.

file_put_contents()
file_put_contents(string $filename, mixed $data, int $flags = 0): int|false
High-level convenience function. Handles open, write, and close automatically. Equivalent to calling fopen(), fwrite(), and fclose() in sequence.
✓ No handle management Overwrites by default
fwrite()
fwrite(resource $handle, string $string, int $length = ?): int|false
Low-level write to an open file handle. Use when you need to write multiple times, interleave reads and writes, or use modes like a with explicit handle control.
Requires fopen() handle ✓ Full mode control

file_put_contents() Flags

Flag ConstantEffectEquivalent to
FILE_APPEND Appends data to the end of the file instead of overwriting Opening with mode "a"
LOCK_EX Acquires an exclusive lock during writing — prevents race conditions with concurrent requests flock($h, LOCK_EX)
FILE_APPEND | LOCK_EX Safe append with locking — the safest choice for log files with concurrent access Mode "a" + flock

Code Examples

file_put_contents() — common patterns
fwrite() — low-level control

Prefer file_put_contents() when…

  • Writing once in a simple operation
  • You want clean, minimal code
  • Appending to a log file (use FILE_APPEND)
  • You don't need interleaved reads

Prefer fwrite() when…

  • Writing multiple times within a loop
  • You need full control over the file handle
  • Writing binary data with specific byte lengths
  • Interleaving reads and writes (mode r+)
⚠ Return Value: Always Check It

file_put_contents() returns the number of bytes written, or false on failure (permissions error, disk full, etc.). Never assume success.

  • If it returns false, your data is lost silently unless you check
  • Disk-full errors are silent by default — always verify in production code
05 / Advanced

Directories & File Uploads

Creating and scanning directories, plus the critical security requirements for $_FILES handling.

Directory Functions

mkdir()
mkdir(string $directory, int $permissions = 0777, bool $recursive = false): bool
Creates a directory. With $recursive = true, creates all parent directories in the path. Without it, fails if the parent doesn't exist. Use 0755 for web-accessible folders.
Permissions: 0755 typical ✓ recursive = true for deep paths
scandir()
scandir(string $directory, int $sorting = SCANDIR_SORT_ASCENDING): array|false
Returns an array of all files and directories in a folder, always including . (current) and .. (parent). You must filter these out in almost all real-world uses.
⚠ Always includes . and .. ✓ Use array_diff to filter
is_dir() / file_exists()
is_dir(string $path): bool
Always check before creating directories to avoid errors. is_dir() confirms it's a directory specifically; file_exists() checks for files or directories.
✓ Use before mkdir() Prevents redundant errors

The $_FILES Superglobal

When a user submits an HTML form with enctype="multipart/form-data", PHP populates $_FILES.

$_FILES["avatar"] = [
    "name" => "profile.jpg", // original filename (user-supplied — UNTRUSTED)
    "type" => "image/jpeg", // MIME type — UNTRUSTED (set by browser)
    "tmp_name" => "/tmp/php7hGbFs", // temp path on server — safe to use
    "error" => 0, // UPLOAD_ERR_OK = 0 means success
    "size" => 48392, // bytes — verify against your max
];

Upload Security Checklist

  • ✓
    Check $_FILES['x']['error'] === UPLOAD_ERR_OK first. Never process an upload without verifying the error code is 0.
  • ✗
    Never trust $_FILES['x']['type']. The MIME type is sent by the browser and can be forged. Use PHP's finfo_file() to detect the true type server-side.
  • ✓
    Use is_uploaded_file($tmp) before move_uploaded_file(). Ensures the file came through an HTTP upload (not a local path injection).
  • ✓
    Use move_uploaded_file(), never copy() or rename(). It handles the atomic move and also implicitly calls is_uploaded_file().
  • ✗
    Never use $_FILES['x']['name'] directly as a path. Sanitize with basename() and generate your own filenames (e.g., uniqid()) to prevent path traversal.
  • ✓
    Validate file extension from your allowlist. Combine with finfo_file() verification and reject anything that doesn't match both.
  • ✓
    Store uploads outside the web root or disable PHP execution in the upload directory to prevent uploaded PHP scripts from running.
